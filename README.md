# Домашнее задание к занятию «Базы данных, их типы» - Выполнил: Тимохин Максим Викторович

### Задание 1. СУБД

### Кейс
Крупная строительная компания, которая также занимается проектированием и девелопментом, решила создать 
правильную архитектуру для работы с данными. Ниже представлены задачи, которые необходимо решить для
каждой предметной области. 

Какие типы СУБД, на ваш взгляд, лучше всего подойдут для решения этих задач и почему?

1.1. Бюджетирование проектов с дальнейшим формированием финансовых аналитических отчётов и прогнозирования рисков.
СУБД должна гарантировать целостность и чёткую структуру данных.

1.1.* Хеширование стало занимать длительно время, какое API можно использовать для ускорения работы?

---
### Ответ 1.1

Лучше всего подойдут реляционные СУБД (например: PostgreSQL, Oracle, Microsoft SQL Server). Они гарантируют целостность и чёткую структуру данных за счёт строгой схемы, ограничений (foreign keys, check), соответствия ACID. Это критично для финансовых операций, где недопустимы потеря или дублирование данных, и где требуется сложная аналитика с соединениями и агрегациями.

Ускорение хеширования*
Если хеширование (например, при построении хеш-индексов или контрольных сумм) стало узким местом, можно использовать аппаратно-ускоренные библиотеки через их API:

- Инструкции современных процессоров (CRC32, SHA-1/SHA-256 через Intel SHA Extensions) с вызовом из С/С++ через _mm_crc32_u64 или OpenSSL's EVP.

- Для некриптографических хешей – API библиотек xxHash или Google FarmHash, дающих высокую скорость.

- Если говорить о БД, то можно перевести часть вычислений на GPU через CUDA API (например: при пакетной проверке целостности).

- В PostgreSQL можно использовать расширение pg_hash или встроенную оптимизацию с аппаратным CRC для WAL и индексов.

---

1.2. Под каждый девелоперский проект создаётся отдельный лендинг, и все данные по лидам стекаются в CRM к 
маркетологам и менеджерам по продажам. Какой тип СУБД лучше использовать для лендингов и для CRM? 
СУБД должны быть гибкими и быстрыми.

1.2.* Можно ли эту задачу закрыть одной СУБД? И если да, то какой именно СУБД и какой реализацией?

---
### Ответ 1.2

Лендинги и CRM для лидов
- Для лендингов — документо-ориентированная NoSQL (MongoDB): обеспечивает гибкую схему для разных полей каждого проекта и высокую скорость записи/чтения лидов.
- Для CRM — подойдёт либо реляционная СУБД с гибким JSON-типом (PostgreSQL + JSONB), либо та же документная СУБД (MongoDB) при невысоких требованиях к жёсткой структуре клиентов. Компромисс: единая реляционная СУБД с поддержкой JSON даёт и ACID для сделок, и гибкость для хранения неструктурированных данных с лендингов.

Можно ли одной СУБД?*
Да, использовать PostgreSQL. Реализация:

Реляционные таблицы для сущностей CRM (контакты, компании, сделки).

Тип JSONB для хранения специфичных полей лидов с лендингов, где формат может меняться от проекта к проекту.

Индексация JSONB-полей и возможность строить агрегирующие запросы смешанного типа. Такой подход сочетает гибкость NoSQL и надёжность SQL.

1.3. Отдел контроля качества решил создать базу по корпоративным нормам и правилам, обучающему материалу 
и так далее, сформированную согласно структуре компании. СУБД должна иметь простую и понятную структуру.

1.3.* Можно ли под эту задачу использовать уже существующую СУБД из задач выше и если да, то как лучше это 
реализовать?

---
### Ответ 1.3

База знаний по корпоративным нормам и обучению
Нужна простая и понятная структура — реляционная СУБД (PostgreSQL, MySQL). Таблицы «подразделения», «категории», «документы» с иерархическими ссылками (parent_id). Для отображения древовидной структуры компании хорошо подойдут рекурсивные запросы (WITH RECURSIVE). Схема прозрачна, легко сопровождается и поддерживает контроль версий метаданных.

Использовать ли существующую СУБД?*
Да, если в п.1.2 выбран PostgreSQL. Лучшая реализация:

- Добавить схему knowledge_base в ту же БД.
- Создать таблицы department, article, tag.
- Связь с оргструктурой через внешний ключ к таблице сотрудников/подразделений CRM.
- Индексация полнотекстовым поиском (GIN) для быстрого поиска по материалам.
  
Это снизит зоопарк СУБД и упростит администрирование.

1.4. Департамент логистики нуждается в решении задач по быстрому формированию маршрутов доставки материалов 
по объектам и распределению курьеров по маршрутам с доставкой документов. СУБД должна уметь быстро работать
со связями.

1.4.* Можно ли к этой СУБД подключить отдел закупок или для них лучше сформировать свою СУБД в связке с СУБД 
логистов?

---
### Ответ 1.4

Логистика: маршруты и распределение курьеров
Лучше всего графовая СУБД (Neo4j). Она оптимизирована для быстрых операций со связями (поиск кратчайшего пути, объезд точек, учёт дорожной обстановки). Модель данных — узлы (адреса, склады, курьеры, объекты) и рёбра (дороги, назначения). Позволяет эффективно считать маршруты даже при большом количестве точек.

Подключать ли отдел закупок?*
Графовая СУБД может моделировать и цепочки поставок (поставщики, материалы, договоры), но закупкам часто нужны транзакции и отчёты с джойнами. Оптимально — собственная реляционная СУБД для закупок, связанная с графовой через асинхронную шину (Kafka) или CDC. Альтернатива — мультимодельная СУБД вроде ArangoDB, которая умеет и в графы, и в документы, но для критичных отчётов реляционная БД всё ещё надёжнее.

---

1.5.* Можно ли все перечисленные выше задачи решить, используя одну СУБД? Если да, то какую именно?

---
### Ответ 1.5

Все задачи одной СУБД*
Полноценно закрыть все задачи одной системой сложно: финансовой аналитике нужны строгие ACID-транзакции, логистике — быстрые графовые запросы. Максимально близка PostgreSQL с расширениями (например, pgRouting для графовых маршрутов, JSONB для гибкости, полнотекстовый поиск). Но на чисто графовых нагрузках она уступает Neo4j. Если допустим компромисс, PostgreSQL сможет покрыть 1.1–1.3 и частично 1.4, но для высоконагруженной логистики лучше оставить специализированную графовую БД.

### Задание 2. Транзакции

2.1. Пользователь пополняет баланс счёта телефона, распишите пошагово, какие действия должны произойти для того, чтобы 
транзакция завершилась успешно. Ориентируйтесь на шесть действий.

2.1.* Какие действия должны произойти, если пополнение счёта телефона происходило бы через автоплатёж?

---
### Ответ 2.1

Пополнение баланса телефона:

1. Начало транзакции – открывается явная транзакция с уровнем изоляции, гарантирующим целостность.
2. Проверка баланса – чтение текущего остатка на банковском счёте (или карте) с блокировкой строки (SELECT … FOR UPDATE).
3. Проверка достаточности – если остаток меньше суммы пополнения, транзакция откатывается (ROLLBACK).
4. Списание средств – обновление записи банковского счёта: баланс уменьшается на сумму.
5. Зачисление на телефон – увеличение баланса в таблице абонентов (UPDATE).
6. Фиксация и уведомление – COMMIT, после чего в журнал транзакций записывается успешная операция и отправляется SMS/уведомление клиенту.

При автоплатеже*
Действия отличаются тем, что инициатором выступает не пользователь, а система по расписанию/событию:

- Проверка порогового значения баланса телефона (если ниже — запуск).
- Использование ранее сохранённого платёжного токена (с разрешением на автоплатёж).
- Процесс тот же (списание с привязанной карты), но перед пополнением может потребоваться повторная проверка токена через платёжный шлюз.
- Транзакция считается успешной только после получения подтверждения от шлюза.
- Дополнительно пишется лог автоплатежа в отдельную таблицу с идентификатором подписки.

### Задание 3. SQL vs NoSQL

3.1. Напишите пять преимуществ SQL-систем по отношению к NoSQL. 

3.1.* Какие, на ваш взгляд, преимущества у NewSQL систем перед SQL и NoSQL.

---
### Ответ 3.1

Пять преимуществ SQL-систем перед NoSQL:

- ACID-транзакции – гарантируют атомарность, согласованность, изоляцию и устойчивость, что жизненно важно для финансового учёта.
- Строгая схема данных – исключает аномалии, обеспечивает целостность через внешние ключи и ограничения.
- Мощный язык запросов SQL – позволяет выполнять сложные соединения (JOIN), агрегации, подзапросы; легко читается аналитиками.
- Развитая экосистема – инструменты резервного копирования, мониторинга, репликации, BI-системы «из коробки» поддерживают реляционные БД.
- Хорошая производительность на сложной аналитике – оптимизаторы запросов десятилетиями оттачивались для эффективной работы с большими объёмами структурированных данных.

Преимущества NewSQL перед SQL и NoSQL*
NewSQL-системы (CockroachDB, YugabyteDB, TiDB) объединяют сильные стороны:

1. Горизонтальное масштабирование при сохранении полных ACID-гарантий – в отличие от традиционного SQL, который обычно масштабируется вертикально.
2. Распределённая архитектура без единой точки отказа – автоматический шардинг и репликация, как у лучших NoSQL.
3. Привычный SQL-интерфейс – разработчикам и аналитикам не нужно переучиваться (отличие от NoSQL).
4. Поддержка гибридной нагрузки (HTAP) – один кластер может обслуживать и OLTP, и аналитические запросы, снижая затраты на ETL.
5. Отказоустойчивость и гео-репликация «из коробки» – автоматическое восстановление после сбоев кластера без потери данных.

### Задание 4. Кластеры

Необходимо производить большое количество вычислений при работе с огромным количеством данных, под эту задачу 
выделено 1000 машин. 

На основе какого критерия будете выбирать тип СУБД и какая модель *распределённых вычислений* 
здесь справится лучше всего и почему?

---
### Ответ 4.1

Критерий выбора типа СУБД:
Ключевой критерий — горизонтальная масштабируемость вычислительной мощности при работе с огромными данными на 1000 машинах. Важны: отсутствие общих узких мест (shared-nothing), способность параллелить запросы на все узлы, устойчивость к падениям машин, линейный рост производительности при добавлении узлов.

Лучшая модель распределённых вычислений
Наиболее подходящая модель — массово-параллельная архитектура (MPP) с поколоночным хранением и векторизованным выполнением, реализованная в системах класса ClickHouse, Greenplum, Vertica. Почему:

- Shared-nothing: каждый узел обрабатывает свой кусок данных независимо, нет конкуренции за общую память.
- Данные партицированы по всем узлам, запросы автоматически распадаются на параллельные подзадачи.
- Колоночное сжатие и векторные операции позволяют эффективно выполнять аналитические вычисления (агрегации, сканирование больших таблиц).
- Отказоустойчивость: потеря нескольких узлов не останавливает систему.

Альтернатива — Spark SQL с Parquet на HDFS (модель MapReduce в памяти), особенно если вычисления разнородны и требуется гибкая обработка с DataFrames. Но специализированные MPP-СУБД обычно быстрее для структурированных отчётов. Таким образом, выбираем MPP + колоночное хранилище как оптимальное решение для 1000-узлового кластера аналитики.
